# حزمة Phase 0 — التعريف التنفيذي لحوكمة نظام Rakez ERP

> **الحالة:** مُجمَّد — لا يُعدَّل إلا بمراجعة معمارية رسمية
> **المرجع:** `config/ai_capabilities.php` · `config/governance.php` · `config/user_types.php`
> **تاريخ الإصدار:** 2026-04-04

---

## 1) الهدف التنفيذي من Phase 0

Phase 0 ليس مرحلة بناء. هو مرحلة **تجميد القرارات المعمارية** قبل كتابة أي سطر كود في Phase 1.

**المخرجات المطلوبة:**

| # | المخرج | الغرض |
|---|--------|-------|
| 1 | جدول تحويل `userRole` الرقمي → اسم الدور | إلغاء أي غموض في هوية المستخدم |
| 2 | القاموس المجمّد للصلاحيات | منع إضافة صلاحيات عشوائية |
| 3 | التصنيف الهرمي للصلاحيات | تحديد ملكية كل صلاحية لقسم محدد |
| 4 | نموذج Visibility / Action / Scope | القاعدة الموحّدة لكل شاشة وزر |
| 5 | مصفوفة حدود الأقسام | ما يدخل Filament وما يبقى خارجه |
| 6 | معايير الجاهزية | شروط لا يُبدأ Phase 1 بدونها |

**القيود المطلقة:**

- لوحة Filament واحدة فقط (`admin`) — للحوكمة والرقابة والموافقات
- لا يُعاد بناء العمليات التشغيلية داخل Filament
- لا تُضاف صلاحيات جديدة بدون مراجعة معمارية
- لا تُفعَّل الصلاحيات المؤقتة في MVP
- لا تُفعَّل أوامر التجاوز القسري (Override / Force) في MVP

## 2) الحزمة الرسمية لتحويل userRole إلى أدوار اسمية

> المصدر: `config/user_types.php`

### 2.1 — الأدوار التشغيلية (Operational Roles)

هذه الأدوار مرتبطة بحقل `users.type` في قاعدة البيانات وتُحدّد الوصول للنظام التشغيلي (API + Blade).

| الرقم | القيمة النصية (`users.type`) | الدور في Spatie | الوصف |
|-------|------------------------------|-----------------|-------|
| 1 | `admin` | `admin` | مدير النظام التشغيلي — يملك كل الصلاحيات عبر `Gate::before` |
| 2 | `project_management` | `project_management` | إدارة المشاريع — صلاحيات إضافية ديناميكية عند `is_manager=true` |
| 3 | `editor` | `editor` | المونتاج والتصوير والتصميم |
| 4 | `developer` | `developer` | المطوّرون العقاريون |
| 5 | `marketing` | `marketing` | قسم التسويق |
| 6 | `sales` | `sales` | موظف مبيعات |
| 7 | `sales` + `is_manager=true` | `sales_leader` | قائد فريق المبيعات |
| 8 | `hr` | `hr` | الموارد البشرية |
| 9 | `credit` | `credit` | قسم الائتمان |
| 10 | `accounting` | `accounting` | قسم المحاسبة (كامل الصلاحيات) |
| 11 | `inventory` | `inventory` | المخزون والوحدات |
| 12 | `user` ← يُحوَّل إلى | `default` | مستخدم بدون قسم محدد |
| 13 | `accountant` | `accountant` | محاسب (صلاحيات جزئية مقارنة بـ `accounting`) |

**قاعدة التحويل في `User::syncRolesFromType()`:**

```
type = 'sales' + is_manager = true  →  role = 'sales_leader'
type = 'user'                       →  role = 'default'
type = أي قيمة أخرى                 →  role = type
```

**ملاحظة حرجة:** الدور `admin` (التشغيلي) يتجاوز كل Gates عبر `AppServiceProvider::Gate::before`. هذا منفصل تماماً عن `super_admin` (الحوكمة).

### 2.2 — أدوار الحوكمة (Governance Roles)

هذه الأدوار لا ترتبط بـ `users.type`. تُمنح يدوياً عبر لوحة Filament وتعمل **بالتوازي** مع الدور التشغيلي.

> المصدر: `config/governance.php` → `managed_panel_roles`

| الدور | الوصف | مستوى الوصول |
|-------|-------|-------------|
| `super_admin` | المتحكم الأعلى — يتجاوز كل فحوصات الحوكمة | كامل |
| `erp_admin` | مدير حوكمة ERP — إدارة المستخدمين والأدوار والصلاحيات + رقابة على كل الأقسام | إدارة + رقابة |
| `auditor_readonly` | مراقب للقراءة فقط — لا يملك أي صلاحية تعديل | قراءة فقط |
| `credit_admin` | مشرف حوكمة الائتمان | رقابة + موافقات الائتمان |
| `accounting_admin` | مشرف حوكمة المحاسبة | رقابة + موافقات المحاسبة |
| `projects_admin` | مشرف حوكمة العقود والمشاريع | رقابة + موافقات المشاريع |
| `sales_admin` | مشرف حوكمة المبيعات | رقابة فقط |
| `hr_admin` | مشرف حوكمة الموارد البشرية | رقابة فقط |
| `marketing_admin` | مشرف حوكمة التسويق | رقابة فقط |
| `inventory_admin` | مشرف حوكمة المخزون | رقابة فقط |
| `ai_admin` | مشرف حوكمة الذكاء الاصطناعي | إدارة قاعدة المعرفة |
| `workflow_admin` | مشرف حوكمة سير العمل والموافقات | رقابة سير العمل |

**شرط الدخول الإجباري:** كل دور حوكمة يجب أن يملك `admin.panel.access` كشرط أول. بدونها لا يدخل اللوحة حتى لو امتلك صلاحيات أخرى.

**القاعدة:** المستخدم يمكن أن يملك دوراً تشغيلياً واحداً + دور حوكمة واحد أو أكثر. `syncRolesFromType()` يحافظ على أدوار الحوكمة عند تحديث الدور التشغيلي.

## 4) النموذج الرسمي لـ Visibility / Action / Scope

كل شاشة أو زر أو عنصر في لوحة Filament يخضع لثلاث طبقات متتالية. الطبقات تُفحص بالترتيب — الفشل في أي طبقة يعني الرفض الكامل.

### 4.1 — تعريف الطبقات

```
┌─────────────────────────────────────────────────────────────┐
│  الطبقة 1: Visibility (هل يرى المستخدم هذا العنصر؟)       │
│  ─────────────────────────────────────────────────────────  │
│  • admin.panel.access → شرط أول إجباري                     │
│  • FilamentNavigationPolicy → هل مجموعة التنقل مرئية؟      │
│  • ChecksFilamentNavigationGroupGate → هل canAccess()؟      │
│                                                             │
│  الفشل هنا = العنصر لا يظهر أصلاً                         │
├─────────────────────────────────────────────────────────────┤
│  الطبقة 2: Action (هل يستطيع تنفيذ هذا الإجراء؟)          │
│  ─────────────────────────────────────────────────────────  │
│  • HasGovernanceAuthorization → GovernanceAccessService      │
│  • canGovernance($permission) لكل زر/إجراء                  │
│  • HasReadOnlyGovernanceResource → قفل CRUD                 │
│                                                             │
│  الفشل هنا = يرى العنصر لكن الزر معطّل أو مخفي           │
├─────────────────────────────────────────────────────────────┤
│  الطبقة 3: Scope (على أي بيانات يُطبَّق؟)                 │
│  ─────────────────────────────────────────────────────────  │
│  • فلترة على مستوى Query (الفريق، القسم، المشروع)         │
│  • Policies في Laravel تحدد نطاق البيانات                  │
│  • الخدمات (Services) تطبّق القيود النهائية                │
│                                                             │
│  الفشل هنا = يرى الشاشة لكن بدون بيانات                   │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 — مسار الفحص الكامل (لكل طلب)

```
المستخدم يطلب صفحة/إجراء في /admin
        │
        ▼
[1] هل يملك admin.panel.access؟ ──── لا ──→ رفض (403)
        │ نعم
        ▼
[2] هل هو super_admin؟ ──── نعم ──→ تجاوز كل الفحوصات
        │ لا
        ▼
[3] هل يملك دور حوكمة مُدار؟ ──── لا ──→ رفض
        │ نعم
        ▼
[4] هل مجموعة التنقل مرئية له؟ ──── لا ──→ العنصر لا يظهر
    (FilamentNavigationPolicy)
        │ نعم
        ▼
[5] هل يملك صلاحية العرض/الإجراء المحدد؟ ──── لا ──→ مخفي/معطّل
    (canGovernance / canViewAny)
        │ نعم
        ▼
[6] هل المورد للقراءة فقط؟ ──── نعم ──→ عرض فقط (لا CRUD)
    (HasReadOnlyGovernanceResource)
        │ لا
        ▼
[7] تنفيذ مع تدقيق (GovernanceAuditLogger)
```

### 4.3 — الآليات التقنية (Traits)

| الآلية | الطبقة | الملف |
|--------|--------|-------|
| `ChecksFilamentNavigationGroupGate` | Visibility | `app/Filament/Admin/Concerns/` |
| `HasGovernanceAuthorization` | Action | `app/Filament/Admin/Concerns/` |
| `HasReadOnlyGovernanceResource` | Action (قفل) | `app/Filament/Admin/Concerns/` |
| `FilamentNavigationPolicy` | Visibility | `app/Services/Governance/` |
| `GovernanceAccessService` | Visibility + Action | `app/Services/Governance/` |
| `GovernanceAuditLogger` | تدقيق | `app/Services/Governance/` |

### 4.4 — القاعدة الذهبية

> **كل شاشة أو إجراء جديد يُضاف مستقبلاً يجب أن يستخدم واحداً على الأقل من هذه الـ Traits.**
> **لا يُسمح بأي Resource أو Page في `app/Filament/Admin/` بدون طبقة Visibility + Action.**

## 5) حدود كل قسم

### 5.1 — مصفوفة الحدود الكاملة

| القسم | داخل Filament (حوكمة/رقابة) | خارج Filament (تشغيلي) | مؤجَّل لما بعد MVP | ممنوع في MVP |
|-------|----------------------------|----------------------|-------------------|-------------|
| **النظام / الإدارة** | إدارة المستخدمين، الأدوار، الصلاحيات، الصلاحيات المباشرة، لقطة الوصول الفعلي، سجل التدقيق، لوحة الحوكمة الرئيسية | — | الصلاحيات المؤقتة (`GovernanceTemporaryPermissionResource`) | لوحات Filament إضافية، أوامر تجاوز قسري |
| **الائتمان** | عرض الحجوزات (قراءة) + إجراءات اعتماد (تمويل، رفض، نقل ملكية، مطالبات) | كل العمليات اليومية عبر API: إدخال حجوزات، تواصل عملاء، خطط سداد | — | CRUD حجوزات في Filament |
| **المحاسبة / المالية** | عرض الإيداعات والوحدات المباعة (قراءة) + إجراءات اعتماد (عمولات، رواتب، دفعات) | كل العمليات اليومية عبر API: إنشاء إيداعات، تسجيل مدفوعات، فواتير | — | CRUD مالي في Filament |
| **العقود / المشاريع** | عرض العقود والوحدات والوسائط (قراءة) + اعتماد طلبات المشاريع الحصرية | كل العمليات عبر API: إنشاء عقود، رفع وسائط، إدارة فرق | — | CRUD عقود في Filament |
| **المبيعات** | عرض الحجوزات والمستهدفات والحضور (قراءة فقط) | كل العمليات عبر API: حجز وحدات، إنشاء حجوزات، جداول عمل | — | أي إجراء تنفيذي في Filament |
| **الموارد البشرية** | عرض عقود الموظفين والأداء والإنذارات والفرق (قراءة فقط) | كل العمليات عبر API: إدارة موظفين، إنذارات، تقييم | — | أي إجراء تنفيذي في Filament |
| **التسويق** | عرض خطط التسويق والمهام والعملاء المحتملين والمشاريع (قراءة فقط) | كل العمليات عبر API: إنشاء خطط، إدارة حملات، إدارة إعلانات | — | أي إجراء تنفيذي في Filament |
| **المخزون** | عرض الوحدات (قراءة فقط) | كل العمليات عبر API | — | أي إجراء تنفيذي في Filament |
| **الذكاء الاصطناعي / المعرفة** | عرض سجلات التفاعل والتدقيق (قراءة) + إدارة قاعدة المعرفة (CRUD) | استخدام المساعد عبر API | — | — |
| **سير العمل / الطلبات** | عرض مهام سير العمل والإشعارات (قراءة) + مركز الموافقات | كل العمليات عبر API | — | — |

### 5.2 — القاعدة الفاصلة

```
داخل Filament = رقابة + قراءة + موافقات/رفض فقط
خارج Filament = كل العمليات التشغيلية اليومية (API + Services + Policies + Blade)
```

**الحالة الوحيدة المسموح فيها بـ CRUD كامل داخل Filament:**
- `UserResource` — لأن إدارة المستخدمين هي عملية حوكمة بطبيعتها
- `RoleResource` — لأن إدارة الأدوار هي عملية حوكمة
- `DirectPermissionResource` — لأن الصلاحيات المباشرة استثنائية
- `AssistantKnowledgeEntries` — لأن قاعدة المعرفة لا علاقة لها بالعمليات

**كل ما عدا ذلك:** قراءة فقط (`HasReadOnlyGovernanceResource`) أو إجراءات اعتماد محددة.

### 5.3 — ترتيب الإطلاق الحسّاس

| المرحلة | الأقسام | المتطلب السابق |
|---------|---------|---------------|
| **الأولى (Phase 1)** | النظام / الإدارة | اعتماد Phase 0 |
| **الثانية** | الائتمان | نجاح Phase 1 + اختبارات |
| **الثالثة** | المحاسبة / المالية | نجاح الائتمان |
| **الرابعة** | العقود / المشاريع | نجاح المحاسبة |
| **التوسع المتحكَّم** | المبيعات، HR، التسويق، المخزون، AI، سير العمل | نجاح المراحل الحرجة |

> **لا يُفتح قسم جديد حتى ينجح القسم السابق في اختبارات الوصول والتدقيق.**

### 5.4 — آلية التدرج المقترحة

يجب إضافة مفتاح `governance.enabled_sections` في `config/governance.php`:

```
'enabled_sections' => [
    'Overview',
    'Access Governance',
    'Governance Observability',
    // يُضاف تدريجياً:
    // 'Credit Oversight',
    // 'Accounting & Finance',
    // 'Contracts & Projects',
    // ...
],
```

`ChecksFilamentNavigationGroupGate` يفحص هذه القائمة قبل أي فحص صلاحيات.
