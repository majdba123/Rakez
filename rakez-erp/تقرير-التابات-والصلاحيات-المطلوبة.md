# تقرير إحصائي: التابات والصلاحيات المطلوبة لتشغيل الـ API دون 403

**الغرض:** حصر كل التابات (الأقسام/الشاشات) وما تستدعيه من مسارات API والصلاحيات المطلوبة — بحيث عند منح تاب لمستخدم نضمن منح كل الصلاحيات **الرئيسية والفرعية** حتى لا يظهر 403 رغم امتلاك السماحية الرئيسة.

**ملاحظة:** التقرير قراءة فقط ولا يتضمن أي تعديل على الكود.

---

## ١. ملخص المشكلة (مثال: المشاريع الحصرية)

- **السيناريو:** عند فتح "طلب مشروع حصري" يلزم اختيار مطور من قائمة (عرض المطورين).
- **الصلاحية الرئيسة:** `exclusive_projects.request` (موجودة لأدوار مثل sales, marketing, credit, accounting).
- **ما يحدث:** الصفحة تستدعي `GET /api/developers` لملء القائمة. هذا المسار **لا يحمل middleware صلاحية** بل يتحقق داخل الـ Controller عبر `ContractPolicy::viewAny(Contract)` الذي يتطلب **`contracts.view` أو `contracts.view_all`**.
- **النتيجة:** أدوار مثل sales, sales_leader, marketing, credit, accounting **لا تملك** `contracts.view` ولا `contracts.view_all` → **403** عند تحميل قائمة المطورين.

**الخلاصة:** تاب "المشاريع الحصرية" يحتاج بالإضافة إلى صلاحيات المشاريع الحصرية صلاحية **فرعية** لـ "عرض قائمة المطورين" (حالياً مضمنة في سياسة العقود: `contracts.view` أو `contracts.view_all`، أو صلاحية مستقلة مثل `developers.view` إن وُجدت).

## ٣. مسارات مشتركة (بدون prefix دور) — والصلاحية الفعلية

هذه المسارات تحت `auth:sanctum` فقط؛ التحقق إما من الـ route أو من الـ Controller/Policy.

| المسار | الطريقة | صلاحية الـ Route | التحقق داخل Controller/Policy | الصلاحية الفعلية المطلوبة |
|-------|---------|-------------------|--------------------------------|----------------------------|
| `GET /project_management/teams/index` | GET | لا يوجد | — | لا يوجد (أي مصادق) |
| `GET /developers` | GET | لا يوجد | `authorize('viewAny', Contract::class)` | **contracts.view** أو **contracts.view_all** |
| `GET /developers/{id}` | GET | لا يوجد | نفس أعلاه | **contracts.view** أو **contracts.view_all** |
| `GET /contracts/index` | GET | لا يوجد | منطق داخل Controller (عقود المستخدم أو حسب الدور) | حسب السياسة |
| `GET /contracts/show/{id}` | GET | لا يوجد | `authorize('view', $contract)` | ContractPolicy::view |
| `POST /contracts/store` | POST | لا يوجد | `authorize('create', Contract::class)` | **contracts.create** |
| `GET /second-party-data/show/{id}` | GET | لا يوجد | `authorize('view', $contract)` | ContractPolicy::view (مثلاً second_party.view للـ sales) |
| `GET /photography-department/show/{contractId}` | GET | لا يوجد | `authorize('view', $contract)` | ContractPolicy::view |
| `GET/PATCH /notifications`... | GET/PATCH | لا يوجد | — | أي مصادق |
| `GET /exclusive-projects` | GET | لا يوجد | منطق في Service (المستخدم يرى طلباته أو الكل إن كان موافقاً) | — |
| `POST /exclusive-projects` | POST | لا يوجد | StoreExclusiveProjectRequest: أي مصادق | — |
| `GET /my-tasks`, `POST /tasks`... | GET/POST | لا يوجد | — | أي مصادق (قد يكون هناك صلاحية tasks.create في مكان آخر) |
| `GET /tasks/sections`, `GET /tasks/sections/{section}/users` | GET | لا يوجد | — | أي مصادق |

**استنتاج مهم:** من يستطيع فتح تاب "طلب مشروع حصري" ويحتاج قائمة المطورين **يجب أن يملك** إما `contracts.view` أو `contracts.view_all` (أو صلاحية مخصصة للمطورين إذا أضيفت لاحقاً). الأدوار التي لديها `exclusive_projects.request` دون `contracts.view`/`contracts.view_all`: **sales, sales_leader, marketing, credit, accounting** → معرضون لـ 403 على `GET /developers`.

## ٥. ملخص: صلاحيات يجب تضمينها في التاب (لتجنب 403)

### ٥.١ تاب "المشاريع الحصرية" (طلب مشروع حصري)

- **الصلاحية الفرعية المطلوبة:** إمكانية **عرض قائمة المطورين**.
- **التحقق الحالي:** `DeveloperController` يستخدم `authorize('viewAny', Contract::class)` أي **contracts.view** أو **contracts.view_all**.
- **التوصية (للتقرير فقط):** لكل دور يملك `exclusive_projects.request` (أو يظهر له تاب المشاريع الحصرية) يجب أن يملك أيضاً إحدى: **contracts.view** أو **contracts.view_all**، أو إنشاء صلاحية مستقلة (مثلاً **developers.view**) وربطها بهذا المسار ومنحها لهذه الأدوار.

**أدوار متأثرة حالياً (لديها exclusive_projects لكن بدون contracts.view/view_all):**  
sales، sales_leader، marketing، credit، accounting.

---

### ٥.٢ تاب الائتمان — عرض التمويل

- **المسار:** `GET /api/credit/bookings/{id}/financing`
- **الـ route يطلب:** `permission:credit.financing.view`
- **في الـ config:** لا يوجد `credit.financing.view`، الموجود فقط `credit.financing.manage`.
- **التوصية:** إما إضافة صلاحية **credit.financing.view** في التعريفات ومنحها لدور credit (وadmin)، أو تغيير الـ route لاستخدام **credit.financing.manage** لجميع عمليات التمويل.

---

### ٥.٣ تاب مكالمات AI

- **الـ routes تطلب:** ai.calls.view، ai.calls.initiate، ai.scripts.manage.
- **في الـ config:** يوجد فقط **ai-calls.manage**.
- **التوصية:** إما إضافة الصلاحيات المفصّلة في definitions وربطها بالأدوار، أو توحيد الـ routes على صلاحية واحدة (مثلاً ai-calls.manage).

---

### ٥.٤ تاب الإعلانات (Ads)

- **الـ routes تطلب:** marketing.ads.view، marketing.ads.manage.
- **في الـ config:** لا توجد تعريفات لهما.
- **التوصية:** إضافة **marketing.ads.view** و **marketing.ads.manage** في `ai_capabilities.definitions` وربطها بدور marketing (وadmin حسب الحاجة).

## ٧. خلاصة

- سبب 403 في أماكن رغم "السماحية الرئيسة": الصفحة تستدعي **مسارات فرعية** (مثل قائمة المطورين، أو عرض التمويل، أو الإعلانات) تطلب **صلاحيات أخرى** غير المربوطة ظاهرياً بالتاب.
- لتشغيل الـ API بشكل صحيح ضمن كل تاب يجب **تضمين كل الصلاحيات المطلوبة** لهذا التاب (رئيسية + فرعية) في الدور أو في مجموعة صلاحيات التاب.
- التقرير أعلاه يحصي التابات، المسارات، والصلاحيات المطلوبة، مع تحديد الفجوات (المطورون، credit.financing.view، ai.calls.*، marketing.ads.*) التي ينبغي معالجتها في الإعداد أو في الكود لاحقاً.

تم إعداد التقرير بالاعتماد على: `routes/api.php`، `config/ai_capabilities.php`، `app/Policies/ContractPolicy.php`، `app/Http/Controllers/Contract/DeveloperController.php`، وملفات الـ Controller ذات الصلة — **بدون تعديل أي كود**.
